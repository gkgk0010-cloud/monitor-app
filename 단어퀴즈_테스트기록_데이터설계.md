# 문제풀이 유형 + CCTV(모니터) 테스트 기록 — 데이터 설계

**quiz_type 구분**
- **output** = 기존 문제풀이 (오늘의 연구, 복습 등)
- **input** = 노션 내용 테스트 (단어·시제부사 등)

기존 **answer_logs**·**student_status**와 합쳐서, output/input 기록을 **같은 방식**으로 남기고 모니터에서 같이 보는 방안.

---

## 1. 어떤 데이터로 하는 게 나은가 (결론)

**추천: 기존 `answer_logs` 확장 + `student_status` 그대로 활용**

| 방식 | 설명 | 장점 | 단점 |
|------|------|------|------|
| **A. answer_logs 확장** | `quiz_type` → 'output' / 'input' 구분 | 한 테이블에 기록, 모니터 쿼리 한 곳, CCTV 로직 그대로 재사용 | 컬럼 하나 추가 필요 |
| B. 별도 테이블 | input만 전용 테이블 신설 | 완전 분리 | 모니터에서 두 소스 조회·합침 필요 |

**이유**
- 이미 **answer_logs**에 `student_id`, `created_at`, `tag`, `correct`로 “누가, 언제, 어떤 태그로, 맞았는지”가 있음.
- **student_status**의 `last_answer_result`, `last_answer_at`, `last_answer_tag`는 “방금 푼 문제 결과”만 저장하므로, **단어 앱에서도 문제 풀 때마다 같은 테이블에 insert + student_status upsert**만 하면 CCTV 파란불/빨간불이 그대로 동작함.
- 테스트 기록(오늘 스코어, Worst 3, 타임라인)은 **answer_logs 하나**에서 `quiz_type`으로 필터하면 됨.

---

## 2. answer_logs 확장 (SQL)

Supabase SQL Editor에서 한 번 실행:

```sql
-- output = 기존 문제풀이(오늘의 연구, 복습), input = 노션 테스트(단어 등)
ALTER TABLE public.answer_logs
  ADD COLUMN IF NOT EXISTS quiz_type text DEFAULT 'output';

CREATE INDEX IF NOT EXISTS idx_answer_logs_quiz_type_created
  ON public.answer_logs (quiz_type, created_at DESC);
```

**quiz_type 값**
- **output** = 오늘의 연구, 복습 등 기존 문제풀이  
- **input** = 노션 내용 테스트 (단어·시제부사 등)  
- 나머지(student_id, student_name, created_at, correct)는 기존과 동일.

---

## 2-1. answer_logs의 tag 규칙 (output / input 동일)

**output (오늘의 연구, 복습)**  
- 지금처럼 **문제 단위 태그**가 잘 들어가고 있음 (가정법, 관계사 등).  
- 모니터에서 “어떤 주제로 풀었는지” 구분 가능.

**input (노션 테스트)**  
- **테스트 이름 = 노션 제목**으로 정한다.  
- **tag**에는 그 **노션 제목**(또는 노션 DB/페이지 제목)을 그대로 넣는다.  
- 예: 노션 DB 제목이 `토익 시제부사` → 테스트 이름 = `토익 시제부사`, **tag = `토익 시제부사`**  
- 예: 노션 페이지 제목이 `시제일치 필수암기` → **tag = `시제일치 필수암기`**

이렇게 하면 모니터에서 output은 “문법 태그별”, input은 “노션 테스트(제목)별”로 같은 방식으로 집계·Worst 3·타임라인에 쓸 수 있다.

| 구분 | 테스트 이름 | tag에 넣는 값 |
|------|-------------|----------------|
| output | 오늘의 연구, 복습 등 | 문제 태그 (가정법, 관계사 등) — 기존 유지 |
| input | 노션 제목에 맞춘 테스트 이름 | **노션 제목** (DB 제목 또는 페이지 제목) |

---

## 3. 노션 테스트(단어 앱)에서 넣을 데이터 (1문제당 1행)

노션 내용 테스트에서 **한 문제 풀 때마다** Supabase에 한 번 insert:

| 컬럼 | 값 예시 |
|------|---------|
| student_id | 똑패스에서 넘겨준 학생 ID (또는 이메일 등) |
| student_name | 학생 이름 (표시용) |
| created_at | 기본값 NOW() |
| **tag** | **노션 제목** (해당 테스트가 나온 노션 DB/페이지 제목과 동일) |
| correct | true / false |
| **quiz_type** | **'input'** |

**tag 결정 방법 (단어 앱 구현 시)**  
- 노션 API로 데이터를 가져올 때, 해당 DB/페이지의 **title**을 함께 저장해 두고,  
  문제 풀 때마다 그 **title**을 `tag`에 넣으면 됨.  
- 또는 단어 앱 진입 시 “이번 테스트 이름”을 노션 제목으로 고정해 두고, 그 값을 모든 로그의 `tag`로 사용.

**CCTV(파란불/빨간불)까지 같이 쓰려면**  
단어 앱에서 **answer_logs insert** 한 뒤, **student_status**도 upsert:

- `last_answer_result`: `'correct'` / `'incorrect'`
- `last_answer_at`: NOW()
- `last_answer_tag`: **위와 동일한 노션 제목** (예: `토익 시제부사`)

이렇게 하면 모니터의 기존 로직(최근 N초 이내 정답/오답 → 파란불/빨간불)이 **단어 퀴즈에도 그대로** 적용됨.

---

## 4. 모니터(캡처 화면)에서 쓰는 방식

- **오늘의 스코어 / Worst 3**  
  - 기존: `answer_logs`에서 `student_id`, `created_at` 오늘, `correct` 집계  
  - 변경: `quiz_type = 'output'` → **기존 문제풀이만**, `quiz_type = 'input'` → **노션 테스트만**, 조건 없으면 **전체**로 집계 가능.
- **타임라인(로그)**  
  - `answer_logs` 한 번 조회해서 `quiz_type`으로 output/input 구분 표시 (예: 아이콘 또는 라벨).
- **CCTV**  
  - **student_status**는 그대로 사용. 단어 앱에서 위처럼 upsert만 해주면 추가 작업 없음.

---

## 5. 정리: 어떤 데이터로 하면 좋은지

- **테스트 기록**  
  - **answer_logs 한 테이블**에 `quiz_type`만 추가해서, 문법 + 단어 모두 같은 구조로 저장.
- **CCTV(파란불/빨간불)**  
  - **student_status** 그대로 두고, 단어 앱에서 문제 풀 때마다 **answer_logs insert + student_status upsert**만 하면 됨.

이렇게 하면 “따로 만든 노션 테스트(단어) 앱”이어도 **기존 CCTV·모니터와 같은 DB·같은 규칙**으로 테스트 기록이 남고, 나중에 **output(기존) / input(노션) / 전체**로 필터해서 보면 됨.

---

## 6. quiz_type 정리

| quiz_type | 의미 | 예시 |
|-----------|------|------|
| **output** | 기존 문제풀이 (오늘의 연구, 복습) | 문법 검문소 등 |
| **input** | 노션 내용 테스트 | 단어·시제부사 등 |

**기존에 'grammar'로 들어간 데이터**가 있다면 `supabase_quiz_type_grammar_to_output.sql` 실행해서 한 번에 `output`으로 변경하면 됨.
